Contexto Bancolombia busca fortalecer su equipo de Inteligencia Artificial con ingenieros capaces de diseñar, construir y desplegar soluciones de IA generativa en producción. Esta prueba técnica evalúa su capacidad para construir un sistema RAG (Retrieval-Augmented Generation) end-to-end, desde la adquisición de datos hasta la interacción con el usuario final a través de un agente conversacional.    Importante: Esta prueba no busca una solución perfecta ni un producto terminado. Busca entender cómo piensa, diseña, implementa y documenta soluciones técnicas. Valoramos las decisiones de arquitectura tanto como el código funcional.  2. Objetivo Construir un asistente virtual del Grupo Bancolombia que responda preguntas sobre productos, servicios y contenido publicado en la sección de personas del sitio web, utilizando una arquitectura RAG con un agente conversacional accesible mediante una interfaz de chat.  3. Alcance de la Solución La solución debe cubrir las siguientes capas de principio a fin:  3.1. Adquisición de datos (Web Scraping / Crawling) Implementar un proceso automatizado que: • Recorra https://www.bancolombia.com/personas y descubra los links internos de artículos, productos y páginas informativas. • Descargue el contenido textual de cada página encontrada. • Almacene los datos crudos de forma estructurada con metadatos: URL de origen, título, fecha de extracción, categoría.    No se requiere agotar todo el sitio. Se espera un mínimo de 50 páginas/artículos procesados. El candidato debe documentar sus decisiones sobre profundidad de crawling, manejo de contenido dinámico (JavaScript rendering) y estrategia de respeto a robots.txt.  3.2. Procesamiento y limpieza de datos Implementar un pipeline de limpieza que: • Elimine HTML, navegación, footers, banners y contenido no informativo. • Extraiga el contenido relevante de cada página: texto principal, títulos, subtítulos.

Generación de embeddings y almacenamiento vectorial Implementar el proceso de: • Generación de embeddings a partir del texto limpio utilizando un modelo de embeddings (abierto o comercial). • Almacenamiento en una base de datos vectorial: Pinecone, ChromaDB, Weaviate, Qdrant, OpenSearch, pgvector u otra. • Indexación con metadatos que permitan filtrado posterior: URL de origen, categoría, etc.    El candidato debe justificar: la elección del modelo de embeddings, la dimensionalidad, la base vectorial seleccionada y la estrategia de indexación.  3.4. Retrieval y Generación (RAG) — Servidor MCP El componente de recuperación y generación debe implementarse como un servidor MCP (Model Context Protocol). MCP es un estándar abierto que permite exponer capacidades (tools, resources, prompts) de forma que cualquier cliente compatible pueda consumirlas. Al implementar el RAG como servidor MCP, la base de conocimiento se convierte en una capacidad reutilizable, desacoplada del agente y del frontend.  Tools obligatorias del servidor MCP  Tool Descripción search_knowledge_base Recibe una consulta en lenguaje natural, ejecuta búsqueda semántica contra la base vectorial y retorna los documentos más relevantes con metadatos (URL, título, score de relevancia). get_article_by_url Recibe una URL del sitio de Bancolombia y retorna el contenido completo del artículo indexado correspondiente. list_categories Retorna las categorías disponibles en la base de conocimiento, si el candidato implementó clasificación por categoría durante el scraping. Requisitos técnicos del servidor MCP • SDK oficial: @modelcontextprotocol/sdk mcp con FastMCP (Python) o Java (Sprint IA) • Transporte: soporte obligatorio para stdio. SSE o Streamable HTTP es deseable pero opcional. • Validación de parámetros en cada tool con descripciones claras — estas descripciones son las que el LLM usará para decidir cuándo invocar cada herramienta. • Manejo de errores: base vectorial no disponible, query sin resultados, timeouts, etc.  Flujo RAG dentro de las tools: búsqueda semántica → reranking opcional → retorno de documentos con fuentes (URLs). La construcción del prompt y la invocación del LLM pueden ocurrir en el servidor MCP o en el agente cliente — el candidato debe justificar su decisión.    Modelos aceptados: cualquier LLM accesible por API o ejecutable localmente: Claude, GPT, Gemini, Llama, Mistral, etc. Se puede usar el tier gratuito de cualquier proveedor. Referencia: https://modelcontextprotocol.io · SDKs en https://github.com/modelcontextprotocol https://docs.spring.io/spring-ai/reference/api/mcp/mcp-server-boot-starter-docs.html  3.5. Agente conversacional (Cliente MCP) El agente debe actuar como cliente MCP, conectándose al servidor implementado en el paso anterior para acceder a la base de conocimiento. El agente debe: • Conectarse al servidor MCP e invocar las tools expuestas según la intención del usuario. • Mantener contexto de la conversación: historial de mensajes. • Decidir cuándo consultar la base de conocimiento y cuándo responder directamente (saludos, clarificaciones). • Manejar preguntas fuera de alcance (temas no relacionados con Bancolombia). • Citar las fuentes (URLs) retornadas por el servidor MCP en cada respuesta. • Seguridad para el consumo del agente. • Manejo de las diferentes tipologías de memoria que podría usar el agente (corto, mediano y largo plazo).

Sí, pero la solución debe poder evaluarse sin que el evaluador incurra en costos. Incluya instrucciones para usar alternativas gratuitas o proporcione credenciales temporales si es necesario. ¿Qué profundidad de crawling se espera? Mínimo 50 páginas procesadas. No es necesario agotar toda la sección /personas. Demuestre que el pipeline funciona y que más páginas se podrían agregar fácilmente. ¿Nunca he trabajado con MCP, puedo usar otra cosa para el RAG? No. El servidor MCP es un requisito obligatorio de la prueba. MCP es un estándar abierto con documentación clara y SDKs oficiales en Python, TypeScript y JAVA. La curva de aprendizaje es parte de lo que se evalúa: capacidad de adoptar tecnologías nuevas, leer documentación y construir una implementación funcional. ¿Qué transporte debo usar para el servidor MCP? El transporte stdio es obligatorio (es el más simple y permite probar localmente). Si adicionalmente implementa SSE o Streamable HTTP para consumo remoto, será valorado positivamente pero no es requisito. ¿Puedo usar un LLM local en vez de una API? Sí. Documente los requisitos de hardware y cómo ejecutarlo. Considere que el evaluador puede no tener GPU disponible: incluya una alternativa con API si es posible. ¿Se evalúa el diseño visual del frontend? No se espera un diseño visual elaborado. Se evalúa que sea funcional, claro y que permita una conversación fluida. Streamlit o Gradio son completamente aceptables. ¿Qué pasa si el web scraping falla en algunas páginas? Es esperado. Documente qué páginas no pudieron procesarse y por qué. Se evalúa el manejo de errores y la resiliencia del pipeline, no que logre el 100 % de las páginas. ¿Puedo entregar antes de los 12 días? Sí.
